## ISSUE-001 · 후기 시그널의 감성·부정(negation) 미처리

- **상태**: 🟡 미해결 (추후) — 당장 터지는 버그 아님, 검색 관련성 오염 가능성
- **발견일**: 2026-05-31
- **트랙**: ai/ (signal-fusion / vector-search)
- **우선순위**: 중 (PoC 시연엔 수용 가능, 정식화 전 결정 필요)

### 증상 / 시나리오

네이버 후기 중 이런 **부정 후기**가 있다(실제 표본 발췌, 요약):

> "당화혈색소 건강검진 기준치 넘어가서 3개월 당관리하고 검사받으러 갔는데 …
> 기준치가 잘못됐다며 **검사 거절당함**. … 정말 불쾌했습니다. **다신 안가요**"

병원이 그 서비스를 **거절했고** "안 어울린다"는 내용인데, 파이프라인이 "당화혈색소·건강검진"
같은 토픽을 **긍정 신호처럼** 다뤄 그 병원을 해당 토픽 검색에 노출시키지 않는가 — 가 우려.

### 근본 원인 (코드 판단)

파이프라인 어디에도 **감성(sentiment)·부정(negation) 판정이 없다.** 두 경로로 나뉜다:

**1) 분류 시그널 경로 — 이 예시는 우연히 안전**
- 후기 키워드 추출 [`ai/pipeline/classify.py:288`](../ai/pipeline/classify.py#L288) `_count_medical_keywords`
  는 **고정 화이트리스트** [`_KEYWORD_TO_FOCUS` (classify.py:108)](../ai/pipeline/classify.py#L108) 의
  단어만 센다. 현재 4개 과(피부·정형·이비인후·안과)만 모델링 → "당화혈색소"·"건강검진"(내과 계열)은
  사전에 없어 **추출 자체가 안 됨** → `primary_focus` 0 기여.
- 즉 이 예시가 안전한 건 *감성을 봐서가 아니라* 그 토픽이 사전에 없어서다(우연). 사전에 있는
  단어라면(예: "보톡스 부작용 최악") 부정 후기라도 "보톡스" +1 → "미용 시술" focus 에 투표한다.
  (단, 이 시스템은 "잘한다"가 아니라 "무엇을 **하는가**"를 측정 → 토픽-빈도=focus 는 부분적으로 정당.)

**2) KB 임베딩 경로 — 여기서 우려가 실재**
- [`ai/search/kb_store.py:319-331`](../ai/search/kb_store.py#L319) `build_reviews_chunk` 가
  **후기 본문 원문을 통째로** reviews 청크에 넣어 Titan v2 로 임베딩한다.
- 임베딩은 **감성을 구분 못 한다.** 부정 후기라도 "당화혈색소 건강검진" 토큰이 본문에 있으면,
  사용자가 "당화혈색소 검진 잘하는 곳"으로 검색할 때 **검사를 거절당한 병원이 의미 유사로
  딸려 나올 수 있다.** ← 우려의 실체.
- 가장 고약한 케이스: "이 병원은 X를 안 한다 / 거절했다" 후기가 (X가 사전 단어면) 도리어 X focus
  로 투표하는 **인버전**. negation 감지가 0이라 못 거른다.

### 현재 안전장치 (그래서 당장 치명적이진 않음)

| 안전장치 | 근거 |
|---|---|
| 후기 원문은 화면에 **절대 미표시** (임베딩 전용) | [`kb_store.py:828`](../ai/search/kb_store.py#L828) `_raw_to_search_result` 가 metadata 만 반환 (의료법 §56③) |
| 검색 = 생성형 답변 아닌 **정렬된 목록** — "잘함"이라는 문장 생성 LLM 없음 | [`ai/CLAUDE.md`](../ai/CLAUDE.md) "검색 시점 LLM 0건" |
| 분류 설명(`generate_description`)도 후기 raw 미참조 | [`classify.py:417-419`](../ai/pipeline/classify.py#L417) `ReviewSignal` 은 키워드 빈도·토픽만 보유, 본문 raw 미저장 |

→ "부정 후기가 긍정 평가로 화면에 노출"되는 일은 **없다.** 피해는 **검색 후보 목록에 토픽 유사로
섞이는** 관련성 오염에 한정.

### 해결 방향 (택1 또는 조합)

1. **임베딩 전 후기 감성/거절 게이트** — `build_reviews_chunk` 앞단(또는 `_analyze_reviews`)에서
   강한 부정·"취급 안 함"·"거절"·"안 가요" 류 후기를 **제외하거나 감점**. 정공법.
2. **negation 룰** (저비용) — 토픽 토큰 주변 부정어("안", "못", "거절", "없", "취급 안") 윈도우
   탐지로 해당 토픽 기여 무효화. 1만 룰 베이스라인에도 적용 가능.
3. **LLM 감성 판정** — 비용 때문에 1만 풀커버엔 부적합하나, **LLM 시연 약 500개엔 붙일 만**.

### 결정 필요 사항

- 1만 룰 경로에도 negation 룰(저비용)을 넣을지, 아니면 시연 약 500개만 감성 처리할지.
- 후기 청크에서 부정 후기를 **제외**할지 **감점 보존**할지(임베딩 어휘 풍부도 vs 관련성).
- 처리 시 의료법 재검수 필요: 청크 구성 변경은 [`kb_store.py:247-251`](../ai/search/kb_store.py#L247)
  §56③ 면제 조건과 충돌하지 않는지 `medical-language-reviewer` 검수.

## ISSUE-002 · thin-signal 토픽 retrieval recall — 내과·소아 계열이 top5 미진입

- **상태**: 🟡 미해결 (추후) — 랭킹 버그 아닌 **검색 recall** 한계, 주력강도 도입으로도 안 고쳐짐
- **발견일**: 2026-06-01
- **트랙**: ai/ (vector-search)
- **우선순위**: 중 (주력강도 랭킹 라운드와 별개의 후속 과제)

### 증상 / 시나리오

호흡기·감기 / 예방접종 / 알레르기 등 **내과·소아 계열 thin-signal 토픽**은 해당 진료를
실제로 하는 병원이 있어도 자연어 검색 top5 에 안 뜬다. 이번 라운드에서 relevance 랭킹을
'최고 청크 코사인 1개'에서 **주력 강도(focus intensity)** 로 바꿔 전반 개선을 봤지만
(주력 토픽 84개 A/B: P@1 0.571→0.655, P@5 0.562→0.617, MRR 0.675→0.734), 이 토픽군은
**랭킹을 바꿔도 top5 에 진입조차 못 한다.**

### 근본 원인 (코드 판단)

랭킹(정렬) 문제가 **아니라** retrieval(후보 진입) 문제다. 두 요인이 겹친다:

1. **임베딩 자체가 약함** — 이 토픽군은 병원 사이트·후기에 텍스트가 빈약해서
   (감기·예방접종은 굳이 길게 홍보 안 함) 청크가 짧고 일반적 → Titan v2 임베딩의
   쿼리 코사인이 **~0.41** 수준에 머문다.
2. **min_score 컷에 걸림** — KB Retrieve 후보를 `min_score`(기본 0.42)로 거른다
   ([`kb_store.py:1109`](../ai/search/kb_store.py#L1109) `KB_MIN_SCORE`, 컷은
   [`kb_store.py:1170`](../ai/search/kb_store.py#L1170)·[`kb_store.py:1198`](../ai/search/kb_store.py#L1198)).
   ~0.41 짜리 thin-signal 청크는 **컷라인(0.42)을 못 넘어** 후보 풀에서 탈락 →
   top5 에 들 기회 자체가 없다.

핵심: **주력 강도(`_focus_intensity` [`kb_store.py:957`](../ai/search/kb_store.py#L957))는
컷을 통과한 후보들의 *순서*만 교정한다.** 컷에서 떨어진 청크는 애초에
`_aggregate_by_hospital`([`kb_store.py:915`](../ai/search/kb_store.py#L915))에 안 모이므로,
빈도·청크수 가산이 무의미하다. 그래서 주력강도로도 안 고쳐진다(컷라인을 못 넘어서).

### 현재 안전장치 (그래서 당장 치명적이진 않음)

| 안전장치 | 근거 |
|---|---|
| min_score 는 **무관 쿼리 억지 매칭 방지**가 본래 목적 (0.42 = 92쿼리 eval sweet spot) | [`kb_store.py:1106-1109`](../ai/search/kb_store.py#L1106) 주석 |
| 무관 쿼리는 구조적으로 **'검색 결과 없음'** 으로 떨어짐 (거짓 양성 < 거짓 음성 트레이드오프) | 커밋 `5374431` min-sim 0.3→0.42 |
| 카테고리 경로(`sigungu`·`specialty`)는 KB 미경유 → 내과·소아과 **목록 탐색은 정상 동작** | 검색 경로 이원화 ([`be/api/search.py`](../be/api/search.py)) |
| 위치 검색(bbox+haversine)도 KB lat·lng 기반이라 thin-signal 코사인과 무관 | 검색 경로 이원화 |

→ 즉 "내과 진료 병원을 못 찾는다"가 아니다. **카테고리·위치 탐색으론 다 나온다.**
피해는 **자연어 쿼리에서 thin-signal 토픽이 top5 에 못 드는** recall 한정.

### 해결 방향 (택1 또는 조합)

1. **토픽별 동적 min_score** — 임상/주력 토픽은 0.42 유지, thin-signal 토픽군엔 더 낮은
   임계(예: 0.38)를 허용. 단 무관 쿼리 거짓 양성이 늘 위험 → eval 재스윕 필요.
2. **thin-signal 청크 보강** — standard_specialty·진료과목 메타를 텍스트로 청크에 명시
   주입해 임베딩 신호를 키움 (예: "내과 / 감기 / 예방접종" 키워드 청크). 청크 구성 변경이라
   §56③ 면제 조건 재검수 동반.
3. **하이브리드 retrieval** — 코사인 컷에 걸린 쿼리는 standard_specialty 키워드 매칭으로
   폴백(BoW/룰)해 후보를 보충. KB 임베딩 + 룰 폴백 2단.

### 결정 필요 사항

- min_score 를 토픽별로 가변화할지(거짓 양성 리스크 재평가) vs 청크 보강으로 신호를 키울지.
- 청크 보강 시 의료법 재검수: 청크 구성 변경이 §56③ 면제 조건과 충돌 않는지
  `medical-language-reviewer` 검수.
- 전수 eval: 변경 후 [`be/scripts/_retrieval_eval.py`](../be/scripts/_retrieval_eval.py)(92쿼리)·
  [`be/scripts/focus_rank_eval.py`](../be/scripts/focus_rank_eval.py)(주력 토픽 84개) 무회귀 확인.

## ISSUE-003 · `기타` 25.7% 과적재 → 추론-specialty 검색 교차오염 (무좀→통증클리닉)

- **상태**: 🟠 해결 경로 식별 (HIRA 진료과목 활용신청 대기 / LLM 세부분류 검토)
- **발견일**: 2026-06-05 (사용자 제보)
- **트랙**: be/ (hira_adapter 분류) + ai/ (search specialty 필터)
- **우선순위**: 높음 (데모 검색 신뢰도 직결)

### 증상 / 시나리오

"무좀" 자연어 검색 1위가 **통증·근골격 중심 더건강의원**(액취&무좀 센터를 곁다리 운영). 그 아래도
미용·성형 위주. 정작 무좀을 진료하는 일반 피부과는 밀리거나(아름다운나라 5위) 본문에 "무좀"을 안 적어
(리더스 0회) 안 잡힌다. → "표준 진료과목 너머 실제 주력"이라는 모토와 충돌해 보이는 대표 케이스.

### 근본 원인 (코드+실데이터 판단, 2026-06-05 실측)

두 층이 겹친다 (ISSUE-002 thin-signal 과도 결합):

1. **분류 입도 — `기타`가 25.7%(795/3098, 단일 최대 버킷).** `map_standard_specialty`
   ([`be/adapters/hira_adapter.py:53`](../be/adapters/hira_adapter.py#L53))가 HIRA 종별(clCdNm)+병원명
   파싱에만 의존. `getHospBasisList`는 의원·병원의 진료과목(dgsbjtCdNm)을 null로 줘서, 이름에 양방과
   키워드 없는 미용·통증·성형 부티크가 전부 `기타`로 떨어진다. **분류 3098개 중 무좀/진균/일반 피부질환을
   primary_focus 로 뽑은 병원 0건** (상위 focus 전부 미용·치과·한방).
2. **검색 specialty 필터가 `기타`를 함께 허용.** 추론 specialty(무좀→피부과)에 `[피부과, 기타]`를 적용
   ([retrieve_hospital](../ai/search/kb_store.py))해 분류 못 한 특화 부티크를 배제 않으려는 의도지만,
   `기타`인 통증·성형 클리닉까지 피부과 검색에 새어 들어온다. + thin-signal이라 코사인이 0.42 바닥에
   붙고 `_focus_intensity` 빈도 보너스가 크롤 보일러플레이트(네비 메뉴 "무좀×38" 도배)를 1위로 밀어올림.

### 현재 안전장치

| 안전장치 | 근거 |
|---|---|
| 카테고리·위치 탐색은 standard_specialty 완전일치라 교차오염 없음 | 검색 경로 이원화 |
| 랭킹 reweight/dedup 실험으로 무좀 미해결 확인(노이즈 재배치) | 실측 2026-06-05 |

### 해결 방향 (택1 또는 조합)

1. **HIRA 진료과목 적재** — `MadmDtlInfoService2.8/getDgsbjtInfo2.8`(ykiho별)로 이름파싱 대체. 공식·권위
   있으나 **data.go.kr 세부정보서비스 활용신청 선행 필요**(현재 403 미인가). 덤: `getMedOftInfo`로 의료기기 갭도 채움.
2. **LLM 기반 세부분류** — 공공데이터에만 의존 않고, 이미 가진 크롤 자칭 텍스트를 Haiku/Nova로 분류해
   `기타`를 정형·재활·마취통증 등으로 내림. 활용신청 대기 불필요. 옛 키워드 추론(_infer_standard_specialty)이
   치과/한의원 오분류로 폐기됐던 걸 LLM이 보완. → **"기타를 없애기"가 아니라 "세부 분류"가 목표.**
3. **LLM 질의어 보강** — 동의어 사전(쿼리 확장)이 thin-signal에서 한계(§4-1) → 사용자 자연어를 경량
   LLM(Nova/Haiku)으로 retrieval 최적화 쿼리로 재작성. 단 "검색 시점 LLM 0회" 정체성과 트레이드오프.

### 결정 필요 사항

- 분류 개선을 HIRA 진료과목(활용신청 대기) vs LLM 세부분류(즉시 가능, 비용/정확도) 중 무엇으로 갈지.
- 검색에 경량 LLM 질의어 보강 도입 여부(LLM-0 정체성 포기) — 데모 한정 vs 상시.
- specialty 필터에서 `기타` 허용을 추론 신뢰도에 따라 게이트할지.
