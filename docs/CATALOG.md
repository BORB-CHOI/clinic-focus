# CATALOG — 데이터·신호 실험 기록

> 분류 정확도·신뢰도·검색에 영향을 준 **실측 발견**만 누적하는 로그.
> "어떤 신호를 더하니 / 무엇을 고치니 무슨 효과가 있었나"를 기록한다.
> 중간 스냅샷·폐기된 접근은 남기지 않는다 (설계·결정 근거는 각 트랙 문서/PR에).
>
> **대상**: 강남구 PoC. **원칙**: 우리는 평가하지 않고, 병원이 자기를 어떻게 표현했는지를 보여준다(의료법 §56).

## 1. 다신호 교차검증 — 신호는 *일치할 때만* 신뢰도를 올린다 (핵심 thesis)

병원 자칭(자체사이트) 하나로는 "여러 진료를 다 한다"고 적혀 있어 주력이 안 좁혀진다. 독립적인 외부
신호(블로그·후기·Vision)를 더해 **교차 일치**를 보면 진짜 주력과 신뢰도가 또렷해진다.

설계: 결손(미수집) 신호는 점수에서 **제외**(반값·0 아님), present 신호끼리만 가중 교차. "확실"
등급은 present 신호 **2종 이상**이 같은 focus 를 가리킬 때만(coverage 천장). 그래서 신호가 늘어난다고
무조건 신뢰도가 오르지 않고, **일치할 때만** 오른다.

> **저자(authorship) 분리**: 병원이 직접 쓴 글(자체 블로그 포함)은 자칭(self_claim)으로 귀속하고,
> 블로그·후기 시그널은 **제3자가 쓴 것만** 남긴다 — 같은 저자는 독립 교차검증이 아니므로.

## 2. 자체사이트 노이즈 정제 — 재크롤 없이 본문 56%↓

depth 크롤이 빨아들인 블로그 RSS 아카이브 덤프·에러/준비중·중복 페이지를 페이지 단위로 제거
(`_filter_noise_pages`). 강남 2,133개 raw 재처리(재크롤 0):

| | 이전 | 이후 |
|---|---:|---:|
| 페이지 수 | 26,377 | 11,531 (**56%↓**) |
| blog 페이지 | 16,221 | 3,442 (**79%↓**) |

자칭 신호에서 자기 글이 아닌 인용·아카이브 잡음을 걷어내 임베딩·분류 입력이 깨끗해졌다.

## 3. 블로그 시그널 소스 — 네이버 키워드 검색 ❌ → 카카오 place앵커 ✅ (교차오염 22배 차)

블로그를 "그 병원이 실제로 쓴 글"의 신호로 쓰려면 **호스트 앵커 정밀도**가 관건. 같은 병원 490개로 실측:

| | 네이버 블로그 키워드 검색 | 카카오 place앵커 blog |
|---|---:|---:|
| 동일 URL 교차오염 | 16.78% | **0.75%** (22배 깨끗) |
| 타 병원 동시언급(리스티클) | 50.7% | **11.2%** |

(리스티클 = 여러 병원을 순위·목록으로 나열한 글.) 네이버 키워드 결과는 절반이 옆 병원·명단·구인글이라
호스트 앵커가 못 됨 → **폐기**. place_id 에 묶인 카카오 place앵커를 블로그 시그널의 권위 소스로 채택.

> 교체 트레이드오프(실측): 정제 자체사이트로 분류 커버리지 848→2,138(약 2.5배)로 늘었으나,
> 노이즈로 부풀던 네이버 키워드를 빼고 깨끗하지만 적은 카카오 앵커로 바꾸니 "확실"은 19→10 으로
> 정직하게 하락 — 신호가 깨끗해지는 대가(커버리지·후기축은 4의 네이버 후기 합류로 회복).

## 4. 네이버 플레이스 후기 합류 — 후기 커버리지 5배, 그러나 무조건 ↑는 아니다

로컬 PC 회색지대 크롤(Playwright 18~25초/건)로 받은 네이버 플레이스 후기를 DDB 적재
(`NAVER#PLACE#REVIEWS` **3,085개 / 리뷰 55,532건**). 후기 커버리지 카카오 단독 641 →
카카오∪네이버 **~3,100(약 5배)**.

A/B (네이버 보유 강남 무작위 34, 같은 병원을 후기 **빼고 vs 넣고** 분류):

| 지표 | 결과 |
|---|---|
| 후기 신호 신규 확보(전무→present) | **21/34 (62%)** |
| 신뢰도 점수 | ↑21 / =4 / **↓9** · 평균 **Δ +14.7** |
| 등급 변동 | 정보부족→추정 4 · 추정→확실 1 · 추정→정보부족 3 |

**정직하게**: 순효과는 +지만 **9건(26%)은 하락**. 이유는 버그가 아니라, 네이버 후기가 자칭과 **다른
방향**을 가리키면 교차검증이 "자칭과 후기가 안 맞음"을 드러내 점수를 내리는 것 — 과장 자칭을 후기가
반박하는 케이스다. 신호가 늘수록 *일치할 때만* 신뢰도가 오른다는 1의 설계가 의도대로 작동.

## 5. Vision 하이브리드 — `<img>` 태그가 아니라 *화면 장면을 해석*

초기 `<img>` 태그 URL만 긁는 방식은 로고·히어로배너만 잡혀 **Vision 신호 0**(아무도 "확실" 안 됨).
→ **스크린샷 스크롤(주력) + 필터링한 `<img>`(보조)** 하이브리드로 전환. Sonnet 4.6 이 OCR 이 아니라
페이지 **장면을 해석**한다(scene·detected_procedures·in_image_text). 멀티이미지 배칭으로 병원당
N콜 → 1콜(~28초/개·0크래시, 4GB EC2 스왑+멀티프로세스).

시연 셋 **병원 508곳**(이미지 582장) 분석, 집계는 **병원 단위**: 장면 해석된 병원 **499** · 시술
추출된 병원 **435**(시술 태그 3,218건) · 기기 식별된 병원 241 · 전 이미지가 빈 병원 4. 병원 메인
화면에 게시된 사진에서 라식 장비(WaveLight·LenSx), 임플란트(인비절라인·파노라마 X-ray), 써마지/리쥬란
등이 감지됐다 — 자칭 텍스트엔 없던 신호다.

> 의료법: Vision 이 본 시술/기기는 "병원이 공식 게시한 사진에서 감지됨"으로만 귀속하고, 텍스트 자칭만
> 있는 기기는 Equipment 로 올리지 않는다(권위 출처=Vision 확인·공공 신고만).
>
> 추출 견고성(커밋 `59ce0db`): 데모 508건 중 45건이 `Equipment` ValidationError(LLM 여분 키·source
> Literal 위반)로 SERVICES/DESCRIPTION 가 통째 누락됐다 → `_safe_build` 항목별 검증으로 위반 항목만
> 스킵·나머지 보존. 재실행 시 데모 실패 45→0.

## 6. 룰 태깅 노이즈 제거 — bare 토큰 오매칭 + "기타" 종별 평탄맵

네이버 하락 9건을 원본까지 추적하니, 하락의 상당수는 후기가 옳아서가 아니라 **자칭 태그 자체가
오염**돼 있었다. 두 축:

1. **bare 공통 토큰 오매칭** — 키워드 어휘에 레거시 평탄맵의 한 글자/흔한 토큰("손"·"미용"·"수면")이
   섞여, substring `.count()` 가 **"손님"·"손상"·"미용실"** 에 걸렸다 → 거의 모든 병원이 자칭에
   `손·발(수부외과)` 를 달았다(과 가리지 않고).
2. **"기타" 종별이 평탄맵으로 빠짐** — HIRA 가 `기타` 로 둔 미용/통증 클리닉이 진료과 제약을 못 받아
   4과짜리 옛 스키마로 `[척추, 미용 시술, 무릎·관절, 손·발, …]` 7개 잡탕이 됐다.

**수정**(커밋 `747477a`): 키워드 어휘를 진료과 taxonomy 전용으로(구체 키워드라 오매칭 0) + 룰/LLM 추출기
모두 진료과 제약 taxonomy 로 focus 산출 + "기타"는 전 과목 taxonomy 스캔.

| 지표 (강남 무작위 40, 옛 vs 수정 — 룰 경로) | 옛 | 수정 |
|---|---:|---:|
| **`손·발(수부외과)` 오염 병원** | 10 (25%) | **0** |

정상화 예(익명): HIRA `기타` 로 분류된 한 미용/통증 클리닉의 자칭 focus 가 `[척추,미용,무릎,손·발…7개]`
잡탕 → `[지방흡입·체형,리프팅·탄력,보톡스·필러]` 로 정정. 마취통증·한의 사례에서 진료과 일치 태그로
좁혀지며 신뢰도도 일부 상승(예: 38→47, 37→41).

> **잔여(데모 LLM 경로)**: 위 손·발 10→0 은 룰 경로(use_llm=False, 강남 전체) 기준. 데모(use_llm=True)
> 경로의 LLM 자칭 추출은 747477a 후에도 옛 4과 스키마로 골라 손·발 5건이 남았다 → LLM 의 자칭
> primary_focus 도 taxonomy 산출로 일원화해 0 (커밋 `d12fa75`, keywords·spam_score 는 LLM 값 유지).

> **핵심**: 신뢰도 개선은 **임베딩 수정이 아니라 룰 키워드 태깅 정제**에서 나왔다 — 아래 7 참조.

## 7. 아키텍처 메모 — 무엇이 임베딩되고, 검색은 무엇으로 정렬되나

세 가지를 명확히 분리(코드로 확인):

1. **임베딩 인풋 = 시그널 본문 텍스트** (자칭·블로그·후기·Vision 장면; 광고어 스크럽 + 동의어 주입).
   룰 태그(`primary_focus`)·`confidence_score` 는 **메타데이터**(필터/표시)로 따로 간다 → 6의 태그
   노이즈는 표시·신뢰도에만 영향, **벡터 랭킹은 오염된 적 없음**.
2. **검색 경로 이원화** — 카테고리(`sigungu`+`specialty`)는 DDB GSI 직조회(KB·임베딩 미경유),
   자연어/위치는 KB Retrieve(벡터). 완전히 다른 경로.
3. **랭킹 = 벡터 유사도**(본문 임베딩 ↔ 쿼리), 신뢰도가 아님. 신뢰도는 선택적 필터/정렬용(기본 OFF,
   의료법 차별노출 회피).

> 파이프라인 가드(커밋 `2237be1`): primary_focus 가 빈 병원(신호 부족)은 query_text="" 가 돼
> KB Retrieve 가 InvalidQueryError 로 run_index_pipeline 전체를 실패시켰다(데모 7건) → 같은-주력
> 추천만 건너뛰고 빈 결과로 진행. "신호 없는 병원도 검색에서 사라지지 않게"(1·비차별노출)와 직결.

### 측정의 한계 — 신뢰도 ≠ 검색 품질 (다음 과제)

지금까지 "확실/추정/정보부족"(=신호 합의도)만 봤는데, 이건 **검색 품질의 대리지표가 아니다**.
신뢰도는 per-hospital 신호 일치도이고, 검색 품질은 "쿼리에 관련 병원이 상위에 잘 뜨나"(precision@k·nDCG)다 —
랭킹은 신뢰도가 아니라 본문 임베딩이 좌우하므로 둘은 거의 직교한다. **검색 품질은 아직 측정 안 함.**
→ 다음: 현실적 자연어 쿼리셋 + 약한 정답(진료과/focus)으로 retrieval eval(precision@5·MRR) 구축.
