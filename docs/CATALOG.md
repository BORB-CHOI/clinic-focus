# CATALOG — 데이터·신호 실험 기록

> 신호를 더하거나 고쳤을 때 분류·검색에 **무슨 효과가 있었나**만 누적하는 로그. 강남구 PoC.
> 중간 스냅샷·폐기 접근·코드 디테일은 PR·이슈로. 의료법 §56: 우리는 평가하지 않고,
> 병원이 자기를 어떻게 표현했는지 보여준다.

---

## 왜 "데이터 모아서 RAG"로는 안 되나 (이 로그의 결론)

목표는 **"표준 진료과목 너머, 이 병원이 실제로 무엇에 주력하나"**다. 데이터를 모아 임베딩→벡터검색만
하면 안 나온다 — 난이도가 검색 알고리즘이 아니라 그 **앞뒤**에 있기 때문이다.

1. **원천이 "주력"을 직접 말해주지 않는다.** 자칭은 "다 한다"고 적어 주력이 안 좁혀지고, 블로그·후기는
   제3자 리스티클·잡음, 이미지는 `<img>`론 로고만 잡힌다. → 신호별 **정제 + 저자 분리 + 교차검증**이
   *선행*돼야 주력이 추출된다(§1~§6). 단순 수집·임베딩은 noise-in-noise-out.
2. **표준 진료과목이 "주력"을 못 담는다.** 모발이식·수면·미용 부티크는 HIRA가 `기타`로 떨군다.
   표준 축으로 필터링하면 정작 그 쿼리의 핵심 병원이 사라진다 — 제품의 존재 이유이자 기술적 함정(§7-③).
3. **RAG가 약한 게 아니라 그 주변 가공이 망쳤다.** 검색 미스의 뿌리 3겹은 전부 RAG 자체가 아니라
   전처리/게이팅이었다(§7).
4. **신뢰도 ≠ 검색 품질.** 신뢰도는 per-병원 신호 합의도, 랭킹은 본문 임베딩 — 거의 직교한다. 따로 잡아야 함.
5. **그래서 검색 시점 LLM·에이전틱 RAG는 불필요하다.** 생성 단계 없는 랭킹 검색(Semantic Search,
   검색당 ~$0.00003). AI의 자리는 검색 루프가 아니라 **인제스트 전처리**(청크·분류·메타 정합성)다.
6. **남는 한계는 데이터 품질**이다. thin-signal 병원(1페이지·Vision없음·캐주얼 리뷰)은 알고리즘으로 못 구한다.

## 1. 다신호 교차검증 — *일치할 때만* 신뢰도↑ (핵심 thesis)

자칭 1종으론 주력이 안 좁혀진다. 독립 외부신호(블로그·후기·Vision)의 **교차 일치**로 주력·신뢰도가
또렷해진다. 결손 신호는 점수에서 제외(0 아님), present 2종 이상이 같은 focus를 가리킬 때만 "확실".
신호가 늘어도 무조건 오르지 않고 *일치할 때만* 오른다. (저자 분리: 병원이 쓴 글은 자칭, 제3자 글만 블로그·후기.)

## 2. 자체사이트 정제 — 재크롤 0, 본문 56%↓

depth 크롤이 빨아들인 블로그 RSS 덤프·에러/중복 페이지를 페이지 단위 제거. 강남 2,133개 소급 재처리:
페이지 26,377→**11,531(56%↓)**, blog 16,221→3,442(79%↓). 자칭 임베딩 입력이 깨끗해짐.

## 3. 블로그 소스 — 카카오 place앵커 ✅ (네이버 키워드 ❌, 교차오염 22배 차)

블로그를 "그 병원이 쓴 글" 신호로 쓰려면 호스트 앵커 정밀도가 관건(동일 병원 490개 실측):

| | 네이버 키워드 검색 | 카카오 place앵커 |
|---|---:|---:|
| 동일 URL 교차오염 | 16.78% | **0.75%** (22배) |
| 타 병원 동시언급(리스티클) | 50.7% | **11.2%** |

네이버 키워드는 절반이 옆 병원·명단이라 호스트 앵커 실패 → 폐기. place_id 묶인 카카오 앵커 채택.

## 4. 네이버 후기 합류 — 커버리지 5배 (단, 무조건 ↑는 아님)

네이버 플레이스 후기 적재(3,085곳 / 리뷰 55,532건). 후기 커버리지 641→**~3,100(5배)**.
A/B(무작위 34, 후기 빼고 vs 넣고): 후기 신규확보 **21/34**, 점수 ↑21/=4/**↓9**, 평균 **Δ+14.7**.
↓9건은 버그 아님 — 후기가 자칭과 **다른 방향**이면 교차검증이 과장 자칭을 반박해 점수↓(§1 의도대로 작동).

## 5. Vision — `<img>` 태그가 아니라 *화면 장면 해석* (508병원/582장)

`<img>` URL만 긁으면 로고·배너뿐 → Vision 신호 0. **스크린샷 스크롤(주력)+필터링 `<img>`(보조)**
하이브리드로 전환, Sonnet 4.6이 OCR 아닌 **장면 해석**(scene·시술·기기). 508병원 분석 →
시술 추출 435병원(태그 3,218건)·기기 식별 241병원. 라식 장비·임플란트·써마지 등 자칭 텍스트엔 없던 신호 감지.
(의료법: "병원이 공식 게시한 사진에서 감지됨"으로만 귀속, 텍스트 자칭 기기는 Equipment 미등록.)

## 6. 룰 태깅 노이즈 — bare 토큰 오매칭 (손·발 10→0)

키워드 어휘에 섞인 bare 토큰("손"·"미용")이 substring `.count()`로 "손님"·"손상"에 오매칭 →
거의 모든 병원에 `손·발(수부외과)`가 붙음. `기타` 종별은 진료과 제약을 못 받아 7개 잡탕 focus.
**수정**: 키워드를 진료과 taxonomy 전용 + 룰/LLM 추출기 모두 진료과 제약 + `기타`는 전과목 스캔(빈도 임계).
강남 무작위 40: `손·발` 오염 병원 **10(25%)→0**. (신뢰도 개선은 임베딩이 아니라 **룰 태깅 정제**에서 나옴.)

> **아키텍처 분리**: 임베딩 인풋=시그널 *본문 텍스트* / 룰 태그·신뢰도는 *메타데이터*(필터·표시) → 태그
> 노이즈는 벡터 랭킹을 오염시킨 적 없음. 랭킹=벡터 유사도(신뢰도 아님). 경로 이원화: 카테고리=DDB GSI,
> 자연어/위치=KB Retrieve. 본문 정제는 *크롤 시 1번*, 임베딩 시엔 광고어 스크럽만.

## 7. 검색 품질 — 미스의 뿌리는 3겹, 전부 "RAG 주변 가공"

라이브 자연어 검색에서 미스 발견("M자 탈모"에 치과가 1위, 진짜 모발의원은 누락). 해부하니 RAG 자체가
아니라 전처리/게이팅 3겹이 원인. 라이브 `retrieve_hospital`·`/api/search`로 before/after 검증, LLM 0회.

| 겹 | 원인 | 수정 | 효과 |
|---|---|---|---|
| ① **doc 동의어 도배** | `_enrich_with_synonyms`가 트리거 1회에 동의어 블롭을 청크에 도배 → 변별력 파괴 (19k자 치과에 "탈모" 1회 → "M자 탈모"서 **치과 0.708 1위**) | 주입 제거 + 재인제스트 + min-sim 0.3 | 치과 이탈 |
| ② **query 동의어 도배** | 쿼리 확장 블롭의 **영어 의학용어**(alopecia·telogen effluvium…)가 한국어 코퍼스와 매칭 0 → 쿼리 임베딩을 "일반 의학"으로 끌어 광범위 클리닉을 띄움 | 한글 없는 동의어 컷 + `n_request`=100(리콜 기아 해소) | 모(毛) 전문의원 복귀 |
| ③ **specialty 하드게이팅** ★ | 추론한 `피부과`를 `standard_specialty` 하드필터로 걸어, HIRA가 `기타`로 떨군 모발·수면·미용 부티크를 통째 배제 (모엠 코사인 0.35 최고인데 미등장, 리더스피부과 0.16인데 1위) — **제품 정체성 배반** | 추론 specialty는 `[추론, 기타]` in-필터(명시 선택은 하드 유지) | 코골이→수면의원, 탈모→모발의원 복귀, 정밀도 무회귀(라식→안과·임플란트→치과) |

**검증**: "M자 탈모" `치과 0.708 #1` → **top 전부 모발/피부 의원**(치과·리프팅 0). "임플란트"→전부 치과,
"코골이"→전부 수면의원[기타], "라식"→전부 안과. 181 테스트 통과.

**부수 정리**: 옛 텍스트추론(62% 피부과 오분류) 시절 self_claim 청크가 URL 오매칭 병원에 잔존(강남
189곳/10.8%, KB엔 `피부과`)해 "사마귀"에 성형/한의원이 끼던 stale 메타 → `ingest_hospital(prune_absent)`로
재실행 시 자동 정리(자세히는 [issues/stale-kb-self-claim-metadata.md](issues/stale-kb-self-claim-metadata.md)).

**남은 한계**: thin-signal 병원(예: 모빈치 — 크롤 1페이지·Vision 없음·연예인명 도배 자칭)은 specialty
필터를 풀어도 임베딩이 약해 안 뜬다 → **데이터/청크 품질** 과제(검색 알고리즘 아님).

**다음**: 자연어 쿼리셋 + 약한 정답으로 retrieval eval(precision@5·MRR) 자동화 — 검색 품질을 신뢰도와
별개로 정량 추적.
